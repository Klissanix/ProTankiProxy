# ProTanki — Crystal Overflow Research

## Цель
Проверить гипотезу об integer overflow в обработке TryBuyItem на сервере:
если `count * unit_price` переполняет int32 в отрицательное значение,
сервер вместо списания кристаллов — зачислит их.

---

## Что нашли

### Пакеты покупки (два типа)

| PID | Структура | Используется для |
|---|---|---|
| `-1961983005` | `StringCodec(item_id) + int32(count) + int32(price)` | Краски, оружие, корпуса — count всегда 1 |
| `-523392052` | `StringCodec(item_id) + int32(price)` | Другие предметы (saboteur_m0 и подобные) — без поля count |

Расходники (`pid=-1961983005`) **не тестировались** — не были доступны.
Для расходников count может быть > 1, что делает overflow более применимым.

### Поведение сервера по режимам

| Режим | Что отправляли | Результат | Persisted | Предмет выдан |
|---|---|---|---|---|
| `overflow-count` (краска, price=100) | count=1073741814 | balance=0 | Да | Нет |
| `neg-price` (краска, price=-1000) | count=1, price=-1000 | Откат (400→500 при реконнекте) | Нет | Нет |
| `neg-count` (краска, price=100) | count=-1 | balance=0 | Да | Нет |
| `zero-count` (краска, price=150) | count=0 | balance=0 | Да | Нет |
| `overflow-count` (расходник armor_m0, price=50) | count=2147483628 | balance=0 | Да | Нет |
| `neg-price` (расходник armor_m0, price=-1000) | price=-1000 | Нет CRYSTAL_UPDATE (откат) | Нет | Нет |
| `neg-count` (расходник double_damage_m0, price=50) | count=-1 | balance=0 | Да | Нет |
| `zero-count` (расходник armor_m0, price=50) | count=0 | balance=0 | Да | Нет |

### Выводы по каждому режиму

**overflow-count**
Сервер использует `long` (int64) для умножения, не int32.
`1073741814 × 100 = 107374181400` — не переполняется в int64.
Результат: `500 - 107374181400 < 0` → сервер зажимает в 0.
Гипотеза об int32 overflow **опровергнута**.

**neg-price**
Сервер **валидирует поле price** по своей базе.
При несовпадении — полностью отклоняет транзакцию, нет CRYSTAL_UPDATE.
Клиент показывает списание локально (client-side prediction), при реконнекте откат.

**neg-count / zero-count**
Сервер **не валидирует count на <= 0** до вычисления, но детектирует аномалию после:
- Либо проверяет что `total <= 0` → штраф (balance=0)
- Либо проверяет что `newBalance > oldBalance` → читер → штраф (balance=0)
Предмет не выдаётся. Баланс обнуляется и сохраняется на сервере.

**free (price=0)**
Не протестирован на `pid=-1961983005` — в сессии покупался не тот тип предмета.
По аналогии с neg-price ожидается отклонение (сервер валидирует price).

## Технические детали

### Арифметика сервера
- Умножение `count × price` выполняется в **int64 (long)**, не int32
- Баланс зажимается в 0 снизу: `newBalance = max(0, balance - total)`
- При аномальных значениях (total <= 0) сервер применяет штраф: `newBalance = 0`

### Валидация на сервере
- **price**: проверяется по базе данных сервера — клиентское значение игнорируется или верифицируется
- **count**: не валидируется диапазон (<= 0 проходит), но результат контролируется
- **item_id**: принимается как есть (предмет должен существовать)

### Инструменты
Прокси: `scripts/proxy-crystally/`
- `proxy.py` — MITM с расшифровкой и логированием
- `inject.py` — патчинг TryBuyItem (5 режимов)
- `run.ps1 -InjectMode <mode> -InjectGain <N>` — запуск

Доступные режимы: `overflow-count`, `neg-price`, `neg-count`, `zero-count`, `free`
