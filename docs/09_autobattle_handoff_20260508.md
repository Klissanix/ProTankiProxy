### 1. Состояние AutoBattle

В controls.py добавлены компактные static-поля состояния боя:

- _autobattle_target_
- _autobattle_lost_time_
- _autobattle_seen_x_
- _autobattle_seen_y_
- _autobattle_seen_z_
- _autobattle_lock_time_

При F4 toggle теперь сбрасываются:

- флаг огня
- флаги движения
- target lock
- last seen state
- lost/lock timers
- nav/stuck/reverse state

### 2. Выбор цели

В tank_runtime.py базовый выбор ближайшего врага заменен на более живой visible-first отбор:

- перебор идет по _ptpAutobattleModel_.§521423115512311564123423632234§()
- пропускаются null, self, inactive tanks, союзники и танки с hp <= 0
- проверка союзника остается через tank.§try for const§(this.§5214238124238137123423632234§), то есть DM/FFA с NONE не ломается
- видимость проверяется через компактный helper _ptpAbSee(param1:Vector3)
- среди видимых целей выбирается наименьшая effective distance
- текущая locked цель получает небольшой бонус удержания через уменьшение effective distance

Важно: evaluator priority через BattleService.getCommonTargetEvaluator() сознательно не добавлялся в финальную версию, чтобы не раздувать AS3 и не возвращать FFDec crash. Это потенциальная точка для следующего шага, но сейчас не реализовано.

### 3. Target lock и last seen

Реализовано:

- если цель видима, она сохраняется в _autobattle_target_
- last seen x/y/z обновляется только по видимой цели
- если цель пропала, lock держится короткое время через _autobattle_lost_time_
- если lock протух или цель стала invalid, башня уходит к 0

Текущее окно удержания lock: около 0.85 секунды.

### 4. Наведение и огонь

Сохранена проверенная формула наведения через локальные оси корпуса:

- locX = right * delta
- locY = forward * delta
- aimYaw = atan2(-locX, locY)

Огонь включается только если:

- есть target
- target видим прямо сейчас
- цель жива и не союзник
- дистанция не превышает 100000000 по dist2
- abs(aimYaw) < 0.2

Стрельба не дублируется клавишами. Используется существующий _autobattle_fire_requested_, который уже подхватывается targeting_input.py.

### 5. Связь с движением

Наземная навигация не переписывалась.

Сохранены существующие части:

- raycast candidate offsets
- проверка земли
- обход препятствий
- защита от обрывов
- anti-stuck / reverse behavior
- стремление к центру карты

Добавлен только умеренный bias:

- к видимой цели, если она не слишком близко
- в orbit/lateral режим, если цель уже близко
- к last seen точке, если цель потеряна недавно

Безопасность движения по земле осталась приоритетом над стрельбой.

## Какие проблемы были по дороге

### 1. Нужно было перепроверить факты по AS3, а не верить предположениям

Факты перепроверялись по _ffdec_mine_export:

- tank.§try for const§ действительно не считает NONE-team союзником
- HP читается через tank.§do for final§.§5214231837231850123423632234§
- getCommonTargetEvaluator() в клиенте есть
- BattleService / CollisionDetector / raycast символы совпадают с ранее найденными

### 2. FFDec падал не только из-за размера вставки

Основная отладочная проблема: FFDec несколько раз падал с internal NPE в AVM2SourceGenerator.generateIf / generateForIn.

Изначально это выглядело как типичный constant-pool или "слишком сложная вставка" кейс, но реальная причина оказалась другой:

- в сгенерированном AS3 был сломан if/else блок
- внутрь навигационного участка случайно попал orphaned if(_abCanFire_)
- FFDec маскировал это как internal NullPointerException без нормальной строки ошибки

После удаления этого мусорного if/else сборка восстановилась.

Практический вывод для следующего агента:

- если FFDec снова падает с generateIf/generateForIn без строки и колонки, сначала нужно выгрузить итоговый generated TankEntity AS3 и проверить, не сломана ли структура if/else
- только после этого подозревать constant-pool overflow

### 3. Пришлось ужать боевую логику

По пути пробовались более сложные варианты:

- richer scoring
- evaluator bonus
- более развернутая raycast visibility логика прямо внутри цикла

Они повышали риск FFDec crash и усложняли generated AS3. В итоговой версии оставлена более компактная логика, которая собирается.

## Что обязательно перепроверить следующему агенту

### Ручные боевые сценарии

- DM/FFA: бот должен стрелять по любому другому живому танку
- TDM/CTF/CP/Assault: бот не должен стрелять по союзникам
- Цель за стеной: допускается краткая память last seen, но огонь включаться не должен
- Видимая цель: башня должна доводиться и fire_requested должен включаться только после почти точной наводки
- Несколько врагов: цель не должна дергаться каждый кадр, visible target должен иметь приоритет над невидимым
- Во время движения: танк должен продолжать ехать по земле безопасно, а не ломать nav ради стрельбы
- F4 off: движение, стрельба и lock state должны прекращаться сразу

### Технические точки риска

- Проверить, что helper _ptpAbSee не дает слишком жесткую фильтрацию целей на разных высотах карты
- Проверить, что бонус удержания lock через уменьшение distance не приводит к избыточному туннельному удержанию плохой цели
- Проверить, что orbit bias не заставляет танк лишний раз тереться о стены на узких картах
- Проверить, что _autobattle_lost_time_ и _autobattle_lock_time_ не остаются в странном состоянии после смерти/респавна
- Проверить, что отсутствие evaluator priority не ухудшает заметно поведение против flag carrier в CTF/Assault

## Если следующий агент будет продолжать работу

- Лучше продолжать с текущей компактной версии, а не возвращать большой scoring-блок целиком
- Любые новые AS3-ветвления добавлять малыми шагами и после каждого шага прогонять полную сборку
- Не трогать _ffdec_mine_export напрямую
- Не дублировать стрельбу через keyboard events
- Не использовать Keyboard.isDown() в AS3
- Если понадобится добавить evaluator priority, делать это как очень маленький additive bias и сразу проверять сборку
